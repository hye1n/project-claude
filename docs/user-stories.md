## 2. 결정 사항

> 위 시나리오에서 개발·기획팀이 명시적으로 결정해야 할 항목 5가지

---

### Q1. 누구나 다른 사람의 글을 수정하거나 삭제할 수 있는가?

현재 작성자명은 자유 입력(인증 없음)이다.  
수정·삭제 버튼이 모든 방문자에게 노출되므로, **타인이 김지수의 글을 마음대로 고칠 수 있다.**

- **선택지 A** — MVP 그대로 유지 (누구나 가능 · 1일 내 완료)  
- **선택지 B** — 작성 시 비밀번호 입력, 수정·삭제 시 비밀번호 확인  
- **선택지 C** — 로그인 인증 도입 (v2 범위로 이연)

---

### Q2. 같은 사용자가 같은 글을 새로고침할 때마다 조회수가 올라가는가?

현재 `GET /api/posts/{id}` 호출마다 `view_count + 1`을 무조건 실행한다.  
시나리오 ①·④에서 김지수가 자기 글을 2회 조회하면 조회수 2가 쌓인다.

- **선택지 A** — 현행 유지 (단순·빠름, 조회수 신뢰도 낮음)  
- **선택지 B** — 세션/쿠키 기반 중복 방지 (같은 브라우저 세션 내 1회만)  
- **선택지 C** — IP 기반 중복 방지 (24시간 이내 동일 IP 재조회 무시)

---

### Q3. 검색은 제목과 내용을 동시에 검색하는가, 제목만 검색하는가?

README에는 "제목/내용 검색"으로 명시되어 있으나,  
내용 전체를 LIKE 검색하면 **게시글이 많아질수록 성능이 저하**된다.

- **선택지 A** — 현행 유지 (제목 + 내용 LIKE, MVP 수준에서 허용)  
- **선택지 B** — 제목만 검색으로 축소 (단순·빠름)  
- **선택지 C** — 전문검색(Full-Text Search) 도입 (v2 범위로 이연)

---

### Q4. 글을 삭제하면 즉시 영구 삭제되는가, 복구 가능한 상태로 보관되는가?

현재 `DELETE /api/posts/{id}` 는 DB 행을 즉시 제거한다.  
시나리오에서 김지수가 삭제 후 "잘못 눌렀다"고 되돌리고 싶을 때 방법이 없다.

- **선택지 A** — 현행 유지 (즉시 삭제, 모달로 실수 방지)  
- **선택지 B** — Soft Delete(`deleted_at` 컬럼 추가), 7일 후 배치로 물리 삭제  
- **선택지 C** — 삭제 취소 UI (스낵바 "실행 취소" 버튼, 5초 내 복원)

---

### Q5. 페이지네이션은 클라이언트에서 처리해야 하는가, 서버에서 처리해야 하는가?

현재 전체 글을 한 번에 `GET /api/posts`로 받아 브라우저에서 10개씩 자른다.  
게시글이 1,000건 이상으로 늘면 **초기 응답 지연 및 메모리 부담**이 발생한다.

- **선택지 A** — 현행 클라이언트 페이지네이션 유지 (MVP 범위, 단기 충분)  
- **선택지 B** — 서버사이드 페이지네이션 (`?page=&size=`) 으로 전환 (v2 범위로 이연)  
- **임계 기준 제안** — 게시글 500건 도달 시 B 전환 검토
